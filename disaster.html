Why this tool? The standard gas calculator models observed operations. This tool models what hasn't happened yet — extreme events derived from reading the contract code line-by-line, not from empirical data. Gas drivers from code: (1) sumTreeRemove() in nextProvingPeriod is O(K × log₂N) — K pending removals × depth of Sum Tree. With K=2,000 (contract max) on a dataset with N=871k pieces, this is ~40,000 SLOAD+SSTORE pairs — potentially unprocessable. (2) addPieces has no contract-level limit on batch size K — at K=10,000 a single tx exceeds the entire tipset gas target. (3) dataSetDeleted() callback has a TODO — rails are NOT automatically terminated, they become zombie state. (4) piecesAdded() does NOT update payment rates — that happens only in nextProvingPeriod(). FVM note: cross-contract calls (PDPVerifier → FWSS → FilecoinPay) use CBOR-encoded FVM actor messaging, not EVM CALL — each adds ~20–30M gas overhead beyond the logic itself.

Method Gas components (from code) Scales with Observed max Theoretical worst
createDataSet 6 SSTORE (PDPVerifier state) + BURN_ACTOR call + FWSS callback: EIP-712 verify + 8 SSTORE + 3 FilecoinPay calls (createRail × 1, modifyRailLockup × 1, modifyRailPayment × 1). CDN adds +4 more FP calls. CONSTANT ~250M ~400M (with CDN, 7 FP calls)
addPieces ×K Per piece: sumTreeAdd = h SLOAD + 1 SSTORE (h = trailing zeros of pieceId+1, avg ~1-3) + 2 SSTORE (CID) + 1 SSTORE (leafCount) + 1 SLOAD/SSTORE (totalLeafCount). Fixed: FVM cross-call to FWSS (~25M) + EIP-712 verify + 1 SSTORE/piece in FWSS (metadata). K linear N log ~778M (K=10) No contract limit on K. K=10,000 → ~660 Bgas (unprocessable)
provePossession Per challenge (×5): keccak256 (compute) + findOnePieceId = log₂(N) SLOAD (binary search in Sum Tree) + 1 SLOAD (pieceCid) + MerkleVerify (pure compute, proof is calldata, no SLOAD). Fixed: Pyth oracle call + BURN_ACTOR call + FWSS callback = 2 SSTORE. 5×log₂(N) ~250M (N=871k) ~317M (N=1M). Well-bounded.
nextProvingPeriod (normal) scheduledRemovals read (1 SLOAD) + challengeRange/nextChallenge (2 SSTORE) + FWSS callback: ~5 SLOAD/SSTORE + updatePaymentRates → 1-3 FilecoinPay modifyRailPayment calls. CONSTANT ~130M ~200M (CDN: 3 rail updates)
nextProvingPeriod + K pending removals Per removal: removals.pop() (1 SSTORE) + removeOnePiece: 1 SLOAD + sumTreeRemove = O(log N) iterations each doing 1 SLOAD + 1 SSTORE + delete pieceCid (2 SSTORE) + delete leafCount (1 SSTORE). K × log₂(N) ~9.5B (K=~80, large N) K=2000, N=871k → ~800B+ gas (potentially unprocessable)
schedulePieceDeletions ×K Per piece: 1 SLOAD check + scheduledRemovals.push (1 SSTORE). Fixed: FWSS callback = EIP-712 verify only (no FP calls, no rate update). K linear ~500M (K=50) K=2000 → ~2B gas (but limited by MAX_ENQUEUED_REMOVALS)
deleteDataSet 4 SSTORE (PDPVerifier state zero) + FWSS callback: EIP-712 verify. NOTE: rail termination is NOT done here — it's a separate call to terminateDataSetPayment(). Rails accumulate as zombie state until explicitly terminated. CONSTANT ~200M ~250M
terminateDataSetPayment Calls payments.terminateRail(pdpRailId) + if CDN: terminateRail(cacheMissRailId) + terminateRail(cdnRailId). Each terminateRail = 1 cross-contract FVM call to FilecoinPay + ~3 SSTORE inside. CONSTANT (1 or 3 rails) ~437M ~700M (CDN: 3 rail terminations)
claimDataSetStorageProvider (SP migration) 2 SSTORE (storageProvider, delete proposed) + FWSS callback: storageProviderChanged = 2 SSTORE (info.payee update). No FP calls. Rail payee is NOT updated here — payee in rail stays as old SP until next settle or explicit update. CONSTANT ~200M est. ~250M
